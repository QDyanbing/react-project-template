# 接口 Mock 方案选型

## MSW 2

维护主体：MSW 核心团队和开源社区。

定位：在网络层拦截请求，并通过与服务端接口相同的地址、方法和响应结构提供 Mock 数据。

优点：

- 页面和 Service 继续发起真实网络请求，不需要加入仅用于 Mock 的调用分支。
- 可以按请求地址和方法分别处理接口，并返回状态码、响应头和响应体。
- Handler 可以按业务模块组织，并复用于不同运行环境。
- 未匹配的请求可以直接放行，不影响真实接口联调。

局限：

- 浏览器环境依赖 Service Worker，只能在支持 Service Worker 的安全上下文中运行。
- Mock Handler 仍需要与真实接口类型和响应约定同步维护。
- Service Worker 的注册是异步过程，开发环境需要等待其启动后再渲染应用。

## Vite 开发服务器中间件

维护主体：由项目自行实现和维护。

定位：通过 Vite 插件的 `configureServer` 在开发服务器中注册接口中间件。

优点：

- 请求仍经过 HTTP 接口，不需要修改页面和 Service。
- 可以直接使用 Node.js 能力生成响应。
- 不需要在浏览器中注册 Service Worker。

局限：

- Mock 能力与 Vite 开发服务器绑定，无法直接复用到浏览器外的测试环境。
- 路由匹配、请求体解析、延迟和异常响应等能力需要自行维护。
- 模板需要增加自定义 Vite 插件和服务端中间件代码。

## Service 内部 Mock

维护主体：由项目自行实现和维护。

定位：Service 根据环境直接返回本地数据，不发起网络请求。

优点：

- 实现简单，不需要额外依赖和网络拦截能力。
- 适合只演示静态数据的临时代码。

局限：

- Service 同时包含真实请求和 Mock 分支，职责混合。
- 无法验证请求地址、请求方法、请求参数和统一请求封装。
- Mock 与真实接口走不同调用链，联调时容易出现行为差异。

## 选型结论

选择 MSW 2 作为接口 Mock 方案。

MSW 在网络层拦截真实接口地址，页面、Model、Service 和统一请求层不需要感知 Mock。开发环境与生产环境使用相同的调用链，可以同时验证请求参数、统一响应结构和错误处理行为。

Mock Handler 按业务模块组织，只在明确启用 Mock 的开发环境中生效，生产环境不包含 Mock 能力。未匹配的请求直接放行，可以继续连接真实接口。
